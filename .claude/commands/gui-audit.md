以下のタスクを **gui-auditor エージェント** に依頼してください。

## 引数の解釈

`$ARGUMENTS` が指定されている場合:
- 画面/機能キーワード（例: "ツールバー", "タブ", "プロジェクト読込", "エクスポートのダイアログ"）
  → その領域に絞って監査
- ファイルパス（`src/main/java/juml/app/uml/*.java`）→ そのコンポーネントの UX を監査
- 「使いづらい」など漠然とした不満 → 主要コンポーネントを横断して最優先の問題を洗い出す
- 空欄 → GUI 全体のユーザビリティを俯瞰監査し、重大度の高い問題を優先列挙

## 依頼内容

<task>
対象: $ARGUMENTS

Juml の Swing デスクトップ GUI の使い勝手を、コードを根拠に監査してください:

1. **対象特定**: 不満/対象に対応する画面コンポーネントを Glob/Grep で特定する
2. **導線追跡**: ボタン/メニュー/ツリー操作 → ハンドラ → 実処理の経路を読み、
   フリーズ・無反応・分かりにくさの原因を推定する
3. **根拠付き指摘**: 各問題に「症状（ユーザー視点）」「原因（file:line）」「重大度」を付ける
4. **改善提案**: VS Code タブ中心アーキ（.claude/rules/gui-tab-architecture.md）と
   責務分離に沿った最小の変更案を示す（実装はしない）

特に次を重点確認すること:
- EDT ブロッキング（actionPerformed 内の重処理 → フリーズ）
- アイコンボタンの tooltip 欠落・メニューのショートカット/ニーモニック未設定
- ダイアログの既定ボタン/Esc 未対応・曖昧な検証/エラーメッセージ
- 長時間処理の進捗・キャンセル可否
- 空状態（プロジェクト未読込・タブ 0 件）の案内
- タブ増殖/重複などタブ中心ゴールとの不整合

報告は日本語で、技術用語は英語のまま。ファイル参照は file:line 形式にしてください。
</task>

## 出力フォーマット

```
## GUI ユーザビリティ監査結果

### 対象範囲
- <監査した画面/コンポーネント>

### 指摘事項
#### [重大度] <英語タイトル>: <日本語の症状>
- 症状（ユーザー視点）: <何が使いづらいか>
- 原因: `<file>:<line>` <実装上の理由>
- 改善案: <既存アーキに沿った最小の直し方／目的>

### 優先対応の提案
1. <最優先で直すべき項目と理由>
```
